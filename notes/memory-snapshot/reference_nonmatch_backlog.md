**tools/modern_sweep.sh** compiles every NON_MATCH .c with agbcc + `-DMODERN=1` (activates the MODERN body but keeps the byte-matching agbcc toolchain — NOT `make MODERN=1`, which switches to devkitARM) and counts real instruction diffs vs the expected target. **tools/mdiff.sh `<src.c> <func>`** shows one function's diff (filter cosmetic label lines: `grep -vE "<[A-Za-z_].*\+0x"`). CAVEAT: the sweep reuses basenames so its leftover .o is unreliable for post-hoc re-diff (input.c ×2, util.c ×3, main.c ×2) — always re-diff with mdiff.sh by full path.

**2026-07-14 — two hard-won lessons (verified empirically this session):**

1. **TESTING A MODERN BODY WITH `-DMODERN=1` CAN LIE — TWO WAYS.** (a) A *failed* compile leaves the target symbol absent, so `diff <(dis)<(dis)` compares two EMPTY disassemblies → bogus "0 diffs". (b) Codegen can differ between MODERN=0 and MODERN=1 via include-order / MODERN-guarded headers (e.g. util.c: `ELF_AVABILITY` from cyberelf.h is only pulled in transitively under one config; adding `#include "cyberelf.h"` at the top even changed register allocation!). **Always test a dropped-dual-form at `-DMODERN=0` (the real build mode) AND check the symbol's byte size, not just the disasm.** tools/mdiff.sh now: takes an optional MODERN arg (`tools/mdiff.sh <src> <fn> 0`), GUARDS empty compiles (prints SYMBOL MISSING), and prints `[cur=N exp=M instrs]`. Confirm every candidate with a real `make DEVKITARM=/c/devkitPro/devkitARM compare` before trusting it.

2. **1-3 diff register-coalescing/spread ties RESIST THE PERMUTER, not just hand-idioms.** Ran 4 permuters concurrently (32 cores): readKeyInput (3-diff coalescing) NEVER improved from base 205 in 3000+ iterations; blizzackMode0 (730→610) and CopyBgMap (535→445) crawled but didn't converge; only beetank (copy-pin basin) improved meaningfully (945→540). So "hand the close ties to the permuter" is NOT a reliable win — the randomizer can't reach the exact agbcc allocation any better than C tweaks can. Reserve the permuter for cases with real structural slack (temps/reorders), not pure allocation ties.

**DECODED InitMotionLocation + IsInHazard (2026-07-14) — both structural, one is a NEAR-MATCH:**

- **IsInHazard (src/player.c:3409)** — RECONSTRUCTION SUCCEEDS to **78 vs 78 instr, 25 mechanical diffs**, blocked by ONE LICM hoist. Key: the committed MODERN body caches `struct Hazard* b = HAZARD(z->hazard[i])` (one pointer → agbcc collapses to a single base → 48 instr, wrong structure). The target instead reads fields via **repeated direct indexing** `gOverworld.terrain.objects[n].FIELD`, which agbcc strength-reduces into 3 loop-invariant bases (gOverworld, &objects[0].start.x @+0x1dc, &objects[0].start.y @+0x1e0) + one induction (idx*24). ALSO must **inline w/h** (not `const u32 w=`) so `x-start.x` evaluates before `w` — the pre-computed temp adds pressure and spills `y` (81 instr). With direct-index + inlined-w/h → 78=78. The last 25 diffs are pure register renumbering: agbcc parks `z` in scratch r3 (no calls in loop), freeing a callee-saved to hoist `z+0x15c` (z->hazard base) as a 3rd invariant; the target keeps `z` in r4 and recomputes `z+0x15c` each iter. Pinning z→r4 cascades (44 diffs); `n`-extraction vs inline makes no difference. Hazard struct: {u16 id@0, attr@2, w@4, h@6, Coord start@8}. It's a gOverworld reader → permuter-poisoned. **Working near-match src saved in this note's approach; genuine 1-LICM-hoist tie.**
- **InitMotionLocation (src/entity_motion.c)** — genuinely structural: target computes loop counts at RUNTIME (`__udivsi3` static count = *(u32*)gStaticMotionGraphics/20; dynamic count from gDynamicMotionGraphicOffsets[0]>>2), NOT the constants STATIC_MOTION_COUNT(254)/DYNAMIC_MOTION_COUNT(200); loops run BACKWARDS (agbcc reversal); dynamic loop has a 3-way split (i≤79→0, i≤143→0, else→pal.dst) where two arms coincide. The MODERN body uses constant counts + forward loops. Full rewrite needed with uncertain payoff — deferred.
