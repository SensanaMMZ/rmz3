**Toolchain path gotcha (Bash tool):** the makefile resolves `AS`/`LD`/`OBJCOPY` from `$(DEVKITARM)/bin`, but in the MSYS the Bash tool starts in, `DEVKITARM=/opt/devkitpro/devkitARM` which **does not exist** — the real toolchain is the Windows install at `C:\devkitPro` (MSYS `/c/devkitPro/devkitARM/bin/arm-none-eabi-*.exe`). Pass the override **on the make command line** (env `export` alone does NOT reliably reach the link rule): `make compare DEVKITARM=/c/devkitPro/devkitARM DEVKITPRO=/c/devkitPro NODEP=1`. Same override for single-object builds (`make build/rmz3/src/.../X.o DEVKITARM=... DEVKITPRO=...`). `diff2.sh` also needs `arm-none-eabi-objdump` → run with `DEVKITARM=/c/devkitPro/devkitARM` exported.

The repo already ships the full matching-decomp pipeline — use it, don't hand-roll objcopy/cmp:

- **`tools/diff.sh <FunctionName>`** — runs `tools/objdiff-cli.exe` to diff `expected/build/rmz3/<obj>` (the matching baseline) vs `build/rmz3/<obj>` (current attempt). Prints `% match` + an instruction-level side-by-side. `tools/diff.sh -f <file.c>` for whole-file. Run `make` first. A function showing ~99.99% with only a table-literal `.word` diff is a true match (confirm with the full-ROM `make` sha1).
- **`tools/refresh-expected.sh <file.c|.o>`** — after a committed sha1 match, snapshot the new `.o` into `expected/` so future diffs use it as baseline. Refuses unless `rmz3.gba` matches `rmz3.sha1`.
- **`tools/diff2.sh <Func>`** — independent **second oracle** (committed): devkitARM `objdump --disassemble=<Func>` vs `expected/`, link-reloc-folded, auto-finds the `.o` from `build/rmz3/rmz3.map`. Two disassemblers (it + objdiff) agreeing = strong signal. **CAVEAT:** if the reference function in the inc was `.byte`-encoded (raw byte blob, common for spawn helpers), objdump renders the expected side as `.word 0x...` data (a `$d` mapping symbol) and diff2.sh shows a bogus all-lines DIFFERS even when bytes match. Decode the first few `.byte`s by hand (e.g. `70 B5` = `push{r4,r5,r6,lr}`) to sanity-check, then trust the **full-ROM `make compare`** as authoritative. **diff2.sh ALSO misses trailing dead-byte artifacts** (it folds `.word` diffs): seimeran `FUN_0808f2e4` has a trailing `.byte 0x70,0x47,0x00,0x00` (dead `bx lr`+pad after its literal pool) that clean C can't reproduce — diff2 said MATCH but the fn was 4 bytes short and the **full-ROM SHA FAILED**. ALWAYS `make compare` before committing; if `objdump -t <o> | grep "F .text"` shows a per-fn size off by a few bytes vs `expected/`, suspect a trailing artifact and leave that one as asm. **`tools/asm-differ/` does NOT work on this GBA map** (no "load address"/LMA line; maintainer flagged it unreliable) — use diff2.sh instead.
- **`MATCHING_IDIOMS.md`** (repo root, committed) — the full agbcc match-idiom catalog + intractable taxonomy + permuter recipe + the 68 upstream-portable functions. The canonical write-up; this memory file is the index. Also `DECOMP.md`/`DECOMP_WORKFLOW.md` (toolchain/process) and `tools/split_inc_multi.py` (splits an inc around named fns, handles the `_b` header).
- **decomp-permuter — now wired up.** Files (committed): `tools/permuter_compile.sh` (flag-robust agbcc compile), `permuter_settings.toml` (compiler_command/assembler_command bypass the noisy Windows make dry-run; asm_prelude_file; compiler_type=gcc), `tools/permuter_arm_prelude.inc` (`.syntax unified`/`.thumb`/`glabel` so the target.s assembles), `tools/permuter_base_fix.py` (colon-initializers→`.f =`, `(...)`→`(void)`). `import.py` patched for forward-slash paths (lives in the gitignored nested `tools/decomp-permuter` repo). pycparser is pip-installed; devkitARM/bin must be on PATH for the scorer's `arm-none-eabi-objdump`.
  **Fully working now** (verified end-to-end on gallisni: prunes to just the function, base score 245, search drives candidates to score 10). `import.py` was patched (in the nested gitignored repo) with `_rmz3_fixups` that runs on the preprocessed source *before* pycparser: converts colon-initializers→`.f =`, `(...)`→`(void)`, drops `extern char assertion[...]` (STATIC_ASSERT, fails only because `cpp -undef` diverges from agbcc struct layout; compile-time-only so safe to drop), and drops the INCASM `asm(".include …")` lines (else the whole AI's asm lands in base.o → comparison junk). `permuter_settings.toml` also passes `-DINCASM(x)=`. Run: `export PATH=/c/devkitPro/devkitARM/bin:$PATH; python tools/decomp-permuter/import.py --keep src/foo.c gtarget.s; python tools/decomp-permuter/permuter.py nonmatchings/<F>/`.
  **Fidelity caveat (why gallisni didn't land):** the target.s IS byte-exact (~97 instr/216B); the real blocker is that the permuter optimizes a *pruned* base.c whose codegen for a context-sensitive function can diverge from the full-TU build. Concretely: the near-miss compiles to 99 instr in both (good), but the winning `do{}while(0)` nudge gave 97 instr (a match) in the pruned base.c yet stayed 99 in the real gallisni.c (+4 ROM overflow). So permuter finds for allocation-sensitive functions don't always transfer. Real fix = faithful base.c: run `--no-prune` (full context, but base.c must be made to compile) **and** patch the scorer/objdump.py to extract only the target symbol from the candidate .o (it currently disassembles the whole .o, which is why pruning was needed). **UPDATE — symbol-isolation patch landed** (nested-repo commit `004daa5`): `Scorer`/`objdump()` now take `function=fn_name` and run `objdump --disassemble=<func>`, scoring only the target symbol (body + its literal pool, up to the next sized symbol). Verified: a 1408-line full `.o` scores on just its 97-line `Gallisni_Update`. So `--no-prune` (faithful full-TU codegen) is now viable — base.c no longer has to be pruned to one function. **PROVEN end-to-end:** `Gallisni_Update` was landed with this exact flow (commit `b6d67472`, full ROM SHA1 verified) — the permuter found a `do{}while(0)` nudge that forced a byte into its home register (`lsrs r5` direct), and it transferred to the full build because `--no-prune` made base.c faithful. The reusable recipe (copy matching `.o` as target + `objcopy -x` to strip locals + `--disassemble` isolation; a candidate whose only diff is tail `.word .rodata` table relocs = real match) is in [[project-open-todos]] item 2. Next targets with the same allocation-class problem: `generator_cannon`, `beetank`.
  **The glabel target.s must be BYTE-EXACT** regardless. Building it from raw inc lines (`glabel <Func>` + `sed -n A,B inc | grep -v '^Func:'`) included ~4 trailing bytes, so the permuter optimized to a +4 target and its "match" overflowed the ROM in the full build. Build the target from the expected `.o`'s exact symbol bytes instead. With an exact target, score 0 = real match; a residual score of ~10 whose only diff is `.word <abs> -> .word .rodata` (a table reloc) IS a real match (resolves at link). The permuter *did* crack gallisni's `lsrs r5` materialization that hand-decomp couldn't (via an inserted empty `do{}while(0)`); landing it for real just needs the exact target. Search is still long/uncertain on the hardest ones.

**Upstream is `mmzret/rmz3`** (branch `main`, actively developed, ~509 .c files) — the canonical decomp of the byte-identical ROM. This base (`lyleaigbedion/rmz3` fork) has **diverged**: each side decompiled different subsets, so neither is strictly ahead. Unmatched functions upstream use `NAKED` + inline `asm()` (== NON_MATCH). The genuinely-hard ones (Seimeran, Wormer, MettaurSwim, Shellcrawler, IceBlock, Ripple, PantheonGuardian, Actor*, gallisni, generator_cannon, beetank) are **asm upstream too** — confirmed intractable even for the experts; don't burn time re-deriving them. Fetch upstream real-C bodies via `raw.githubusercontent.com/mmzret/rmz3/main/<path>` to port functions it solved that this base hasn't (watch for signature drift, e.g. `struct Enemy*` vs custom structs). See [[reference-agbcc-match-idioms]].
